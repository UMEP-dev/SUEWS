# Release-time docs sanity check

Rules for verifying that the **published release documentation** carries no
stray development (`.dev`) markers. Run this check whenever cutting a release
(see `.claude/skills/prep-release/SKILL.md`) and whenever revising the docs of
an already-released version.

Motivated by the post-2026.6.5 gap: the released schema was `2026.5` in code,
but `/stable/` showed a "Development Version" banner and two docs artefacts
still cited a dev schema label. Both are user-visible and both passed every
existing gate.

---

## The two things to check

### 1. No "Development Version" banner on the release build

`docs/source/conf.py` adds a "Development Version" announcement banner to every
page when the resolved version string contains `.dev`
(`is_dev_version = ".dev" in git_version_string`). The version comes from
`get_ver_git.get_version_from_git()`, which honours the
`SUEWS_DOCS_VERSION_REF` environment variable.

- The rolling **`latest`** docs (the `rtd` branch, built by `docs-sync.yml`
  with `SUEWS_DOCS_VERSION_REF=origin/master`) **should** show the banner —
  they are a development build. That is correct.
- The **`stable`** / numbered-release docs **must not**. Read the Docs labels a
  version by the git ref it builds; the release tag must point at a docs anchor
  whose HTML was built with `SUEWS_DOCS_VERSION_REF=<release-tag>`, so the
  version resolves clean (e.g. `2026.6.5`) and no banner is emitted.

**How it is produced:** the `release-docs-anchor.yml` workflow
(`workflow_dispatch`, input `release_tag`) rebuilds the HTML with the tag as the
version ref, asserts the built HTML contains no "Development Version" string,
creates an off-master anchor commit, and force-moves the tag onto it. Do **not**
point a release tag at the `rtd` branch HEAD directly — its HTML carries the
dev banner.

**Manual check** (after RTD rebuilds the moved tag):

```bash
# Should print nothing for the stable/release build
curl -fsSL https://docs.suews.io/stable/inputs/forcing-data.html | grep -c "Development Version"
```

### 2. No `.dev` in user-facing schema labels

The released `CURRENT_SCHEMA_VERSION` must be a plain CalVer label (e.g.
`2026.5`, never `2026.5.devN`). This is already enforced for the constant by the
"Release-PR collapse step" in `.claude/rules/python/schema-versioning.md`. But
two docs artefacts derive from it and can lag:

- **The generated schema artefact** `docs/source/inputs/tables/schema.json` is a
  committed file produced by `suews schema export`. It is **not** regenerated by
  `make generate-rst`, so it goes stale unless regenerated explicitly. At
  release time, regenerate it so its `title`, `version`, `$id`, and
  `schema_version` carry the released label:

  ```bash
  suews schema export -o docs/source/inputs/tables/schema.json
  # or: python -m supy.cmd.schema_cli export -o docs/source/inputs/tables/schema.json
  ```

- **Prose that cites a dev label as the "since" version.** A feature that landed
  in, say, `2026.5.dev7` ships to users as `2026.5`; docs must say
  "Since schema 2026.5", not "Since schema 2026.5.dev7".

---

## Keep-vs-fix: do NOT scrub legitimate cycle narration

Not every `.dev` in the docs is wrong. Historical narration of the **collapsed
dev cycle** is correct and must be kept:

- `docs/source/version-history/v<release>.rst`,
  `docs/source/contributing/schema/schema_versioning.rst`, and
  `docs/source/inputs/transition_guide.rst` legitimately say things like
  "collapsing the `2026.5.dev1`..`2026.5.dev14` development cycle".
- `docs/source/installation.rst` TestPyPI examples use package `.dev0`/`.dev1`
  versions — unrelated to the schema.
- The generated `schema.json` description string also narrates the collapse
  ("Released schema for 2026.6.5 (collapsed from the 2026.5.dev1..dev14
  cycle)") — that is the generator's own output and is correct.

Only fix a dev label that is presented as a **current, user-facing schema
version** (a "since X" or the artefact's own `version`/`title`/`schema_version`
fields).

**Scan command** (flags stray labels, excludes legitimate narration):

```bash
grep -rn "schema 2026\.[0-9]*\.dev\|v2026\.[0-9]*\.dev[0-9]\|\"version\": \"2026\.[0-9]*\.dev" \
  docs/source/ | grep -vi "collapsing\|development cycle\|dev1\.\.\|dev1.*dev14"
```

Any line this prints is a stray dev label to fix.

---

## When this runs

- **At release** (`prep-release` skill checklist): regenerate `schema.json`,
  run the scan above, and after the tag is cut run `release-docs-anchor.yml` so
  `stable` is banner-free. See the checklist item and step note in
  `.claude/skills/prep-release/SKILL.md`.
- **When revising released docs** (no new code): make the doc fix on `master`
  (so `latest` and future releases inherit it), let `docs-sync.yml` refresh the
  `rtd` branch, then re-run `release-docs-anchor.yml` for the affected tag.
